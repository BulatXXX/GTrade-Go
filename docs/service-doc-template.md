# Service Doc Template

Шаблон документации для отдельного сервиса.

Рекомендуется хранить такой файл в папке сервиса как `README.md` или `progress.md`.

## Рекомендуемая структура

```md
# <service-name>

Короткое описание назначения сервиса.

## Что делает сервис

- основная ответственность сервиса
- какие данные хранит
- с какими сервисами взаимодействует

## Что уже готово

- готовые сценарии
- готовые хранилища
- готовые интеграции

## Endpoint'ы

- `GET /health` — описание
- `POST /...` — описание

## Что пока не реализовано

- список отсутствующих сценариев
- известные ограничения

## Локальный запуск

```bash
cp deploy/.env.example deploy/.env
make <service>-up
make <service>-logs
```

## Локальное тестирование

Все тесты:

```bash
make <service>-test
```

Интеграционные тесты:

```bash
make <service>-test-integration
```

## Полезные команды

```bash
make <service>-db-up
make <service>-db-down
make <service>-down
```

## Хранилище

- основная таблица 1
- основная таблица 2

## Файлы реализации

- `internal/handler/...`
- `internal/service/...`
- `internal/repository/...`
- `migrations/...`
```

## Рекомендации по заполнению

- держать документ коротким и практичным
- не дублировать общую архитектуру из `docs/architecture.md`
- перечислять только реально работающие ручки
- если ручка placeholder, писать это явно
- рядом с тестовыми командами всегда указывать готовый `make`-таргет
- если сервис требует БД или внешние зависимости, это должно быть видно сразу в разделе запуска

## Минимальный стандарт для каждого сервиса

- описание назначения
- список готовых endpoint'ов
- список незавершенных сценариев
- как поднять сервис локально
- как прогнать тесты
- какие таблицы или внешние зависимости он использует
