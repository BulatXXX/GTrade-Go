# GTrade

## Описание программы

### 1 Общие сведения

GTrade представляет собой многосервисную программу для работы с данными внутриигровых торговых площадок. Программа получает данные из внешних игровых API, нормализует их, хранит локальный каталог предметов, ведет пользовательские списки наблюдения и отправляет уведомления об изменениях.

Программа применяется как backend-контур многосервисного веб-приложения и как единая API-платформа для клиентских и служебных сценариев. Пользовательский интерфейс реализуется отдельным фронтенд-приложением на Angular.

На текущем этапе программа поддерживает следующие внешние источники данных:

- `warframe.market`;
- `EVE ESI`;
- `tarkov.dev`.

Для источника `tarkov` поддерживается режим `game_mode=regular|pve`.

### 2 Функциональное назначение

Программа предназначена для решения следующих задач:

- предоставление единого API для работы с данными разных игровых торговых площадок;
- поддержка поиска предметов и просмотра данных по ним;
- хранение локального каталога и локализаций;
- поддержка пользовательских профилей, watchlist и настроек уведомлений;
- отправка email-уведомлений;
- обеспечение интеграции отдельных доменных сервисов в единую платформу.

Программа обеспечивает:

- регистрацию и аутентификацию пользователей;
- выдачу и обновление JWT;
- сценарии password reset и email verification;
- хранение пользовательского профиля;
- ведение watchlist;
- поиск по локальному каталогу;
- получение карточек предметов;
- получение актуальных рыночных данных;
- хранение истории цен;
- пакетный импорт и синхронизацию каталога;
- отправку email-уведомлений.

### 3 Описание логической структуры

Программа построена по микросервисному принципу. Внешний доступ организован через `api-gateway`, а бизнес-функции распределены между специализированными сервисами.

Основные составные части программы:

- `api-gateway` — единая внешняя точка входа и маршрутизация запросов;
- `auth-service` — регистрация, логин, refresh token, сброс пароля, подтверждение email;
- `user-asset-service` — профиль пользователя, watchlist, preferences, сценарии price alert;
- `catalog-service` — локальный каталог предметов, локализации, история цен, административные фоновые задачи;
- `api-integration-service` — интеграция с внешними игровыми API и нормализация данных;
- `notification-service` — отправка email и хранение outbox;
- `tools/catalog-importer` — CLI-утилита пакетного импорта каталога.

Связи между составными частями программы:

- фронтенд Angular обращается к `api-gateway` как к единой точке входа;
- `api-gateway` маршрутизирует запросы в доменные backend-сервисы;
- `auth-service` при необходимости вызывает `notification-service` для отправки писем;
- `user-asset-service` использует `catalog-service` для проверки и обогащения данных watchlist;
- `api-integration-service` получает данные из внешних API и при необходимости синхронизирует их с `catalog-service`;
- `catalog-importer` пакетно наполняет локальный каталог через API `catalog-service`.

Для реализации программы используются:

- Go;
- Gin;
- PostgreSQL;
- pgx;
- resty;
- zerolog;
- JWT;
- golang-migrate.

Программа использует отдельные PostgreSQL-базы для stateful-сервисов:

- `auth-service` — пользователи и токены;
- `user-asset-service` — профили, watchlist, preferences, scheduler state;
- `catalog-service` — предметы, переводы, история цен, состояние фоновых задач;
- `notification-service` — outbox уведомлений.

`api-gateway` и `api-integration-service` собственной базы данных не имеют.

### 4 Используемые технические средства

Для локального развертывания и интеграционной работы используются:

- Docker;
- Docker Compose;
- OpenAPI / Swagger.

Программа рассчитана на выполнение в контейнеризированной серверной среде с доступом к PostgreSQL и внешним HTTP API игровых источников. Клиентский доступ к программе осуществляется через отдельное веб-приложение на Angular.

### 5 Вызов и загрузка

Загрузка и запуск программы в локальном контуре выполняются как запуск набора сервисов и вспомогательных компонентов в Docker Compose. Внешний доступ к функциям программы осуществляется через `api-gateway`.

Для выполнения служебных операций может использоваться утилита `tools/catalog-importer`, предназначенная для пакетного наполнения локального каталога через API `catalog-service`.

### 6 Входные данные

Входными данными программы являются:

- HTTP-запросы клиентов;
- ответы внешних игровых API;
- данные административных и фоновых запросов;
- параметры конфигурации из переменных окружения.

Состав и формат входных данных определяются OpenAPI-контрактами сервисов, внутренними DTO взаимодействия между сервисами и форматами данных, возвращаемых поддерживаемыми внешними источниками.

### 7 Выходные данные

Выходными данными программы являются:

- HTTP-ответы клиентам через `api-gateway`;
- нормализованные данные о предметах и рыночных предложениях;
- записи локального каталога, локализаций и истории цен;
- данные пользовательских профилей, watchlist и предпочтений;
- email-уведомления и записи outbox;
- служебные результаты административных и фоновых операций.

Состав и формат выходных данных определяются OpenAPI-контрактами сервисов, внутренними форматами обмена между компонентами и форматом уведомлений, формируемых `notification-service`.

## Техническое задание

### 1 Введение

#### 1.1 Наименование программы

GTrade.

Полное наименование: система управления данными внутриигровых торговых площадок для поддержки пользовательских сценариев поиска, анализа цен, ведения watchlist и отправки уведомлений.

#### 1.2 Краткая характеристика области применения

Система применяется в области обработки данных внутриигровых торговых площадок и каталогов предметов онлайн-игр. Система предназначена для получения данных из внешних игровых API, нормализации этих данных, хранения локального каталога и предоставления пользователю и смежным компонентам унифицированного API.

#### 1.3 Объект применения

Объектом применения является процесс работы пользователя с данными игровых предметов, рыночных цен, персональных списков наблюдения и уведомлений об изменениях цен.

Система используется:

- конечными пользователями через отдельное веб-приложение;
- внутренними и внешними клиентами через HTTP API;
- административными и фоновыми процессами для импорта и синхронизации каталога.

### 2 Основания для разработки

Разработка ведется в рамках учебного проекта по дисциплине, связанной с технологиями создания программного обеспечения.

Основаниями для разработки являются:

- потребность в едином API и локальном каталоге предметов для нескольких игровых источников;
- необходимость отделить пользовательские данные, каталог, интеграции, уведомления и аутентификацию по доменным сервисам.

Тема разработки: многосервисная система GTrade для работы с данными внутриигровых торговых площадок.

### 3 Назначение разработки

#### 3.1 Функциональное назначение

Система должна обеспечивать:

- регистрацию, аутентификацию и сопровождение жизненного цикла учетной записи;
- ведение пользовательского профиля, watchlist и настроек уведомлений;
- ведение локального каталога игровых предметов;
- получение и нормализацию данных из внешних игровых источников;
- хранение истории цен и предоставление поисковых сценариев по локальному каталогу;
- отправку email-уведомлений;
- маршрутизацию внешних запросов через единый API gateway;
- пакетный импорт и внутреннюю синхронизацию каталога.

#### 3.2 Эксплуатационное назначение

Система предназначена для эксплуатации:

- как backend-контур многосервисного веб-приложения;
- как API-платформа для отдельного фронтенда, реализованного на Angular;
- как набор сервисов и утилит, запускаемых локально в Docker Compose;
- как основа для дальнейшего расширения аналитических и пользовательских сценариев.

### 4 Требования к программе или программному изделию

#### 4.1 Требования к функциональным характеристикам

##### 4.1.1 Общая архитектура

Система должна включать следующие основные компоненты:

- `api-gateway`;
- `auth-service`;
- `user-asset-service`;
- `catalog-service`;
- `api-integration-service`;
- `notification-service`;
- `tools/catalog-importer`.

##### 4.1.2 Требования к функциям аутентификации и учетной записи

Система должна обеспечивать:

- регистрацию пользователя;
- авторизацию пользователя;
- обновление access token по refresh token;
- запрос на сброс пароля;
- подтверждение сброса пароля;
- запрос на подтверждение email;
- внутренние защищенные методы для получения email-адресов пользователей другими сервисами.

##### 4.1.3 Требования к функциям пользовательских данных

Система должна обеспечивать:

- создание и обновление пользовательского профиля;
- хранение `display_name`, `avatar_url`, `bio`;
- ведение watchlist пользователя;
- включение и отключение уведомлений по элементам watchlist;
- хранение пользовательских предпочтений;
- поддержку режимов уведомлений `daily_digest` и `immediate`;
- выдачу агрегированного представления профиля и watchlist.

##### 4.1.4 Требования к функциям каталога

Система должна обеспечивать:

- хранение локального каталога предметов;
- получение списка предметов;
- поиск предметов по локальному каталогу;
- получение карточки предмета;
- создание, обновление и деактивацию записей каталога;
- хранение локализаций;
- хранение истории цен по дням;
- запуск административных фоновых задач импорта каталога и контроль их статуса.

##### 4.1.5 Требования к функциям интеграции с внешними источниками

Система должна обеспечивать:

- получение runtime-данных по предметам и ценам из внешних игровых API;
- нормализацию данных внешних источников в единый DTO;
- поиск по поддерживаемым источникам там, где это предусмотрено интеграцией;
- синхронизацию metadata из внешних источников в локальный каталог;
- поддержку источников `warframe`, `eve`, `tarkov`;
- поддержку режима `game_mode=regular|pve` для `tarkov`.

##### 4.1.6 Требования к функциям уведомлений

Система должна обеспечивать:

- прием запросов на отправку email;
- сохранение outbox-записей;
- отправку писем через конфигурируемый provider;
- поддержку тестового режима отправки;
- поддержку интеграции с провайдером Resend.

##### 4.1.7 Требования к функциям API gateway

Система должна обеспечивать:

- единую внешнюю точку входа;
- маршрутизацию запросов к доменным сервисам;
- применение middleware идентификации запроса и логирования;
- защиту отдельных групп маршрутов с помощью JWT.

##### 4.1.8 Требования к входным и выходным данным

Входными данными являются:

- HTTP-запросы клиентов;
- ответы внешних игровых API;

- параметры конфигурации из переменных окружения.

Выходными данными являются:
